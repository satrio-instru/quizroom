## 2. VULNERABILITY ANALYSIS

### 2.1 🔴 KRITIK — Hardcoded Admin Password

**File:** `backend/src/managers/UserManager.ts:4`
```typescript
const ADMIN_PASSWORD = "ADMIN_PASSWORD";
```

**Risk:** Password admin di-hardcode langsung di source code. Siapa pun yang mengakses repo dapat melihat dan menyalahgunakan kredensial admin.

**Dampak:**
- Full control atas semua quiz room
- Bisa memanipulasi soal, skor, dan peserta
- Tidak bisa di-rotate tanpa deploy ulang

**Fix:**
```typescript
// Gunakan environment variable
const ADMIN_PASSWORD = process.env.ADMIN_PASSWORD;
if (!ADMIN_PASSWORD) {
  throw new Error("ADMIN_PASSWORD environment variable is required");
}
```

### 2.2 🔴 KRITIK — CORS Wildcard Origin

**File:** `backend/src/managers/IoManager.ts:13-16`
```typescript
cors: {
    origin: "*",
    methods: ["GET", "POST"]
}
```

**Risk:** Mengizinkan koneksi dari domain mana pun. Memungkinkan:
- Cross-site WebSocket hijacking
- CSRF via Socket.IO
- Data exfiltration dari quiz room

**Fix:**
```typescript
cors: {
    origin: process.env.ALLOWED_ORIGINS?.split(",") || ["http://localhost:5173"],
    methods: ["GET", "POST"],
    credentials: true
}
```

### 2.3 🔴 KRITIK — Tidak Ada Autentikasi pada Submit Event

**File:** `backend/src/managers/UserManager.ts:102-114`
```typescript
socket.on("submit", (data) => {
    const userId = data.userId;  // ← userId dari client, bukan dari session
    const problemId = data.problemId;
    const submission = data.submission;
    const roomId = data.roomId;
    // ...
    this.quizManager.submit(userId, roomId, problemId, submission)
});
```

**Risk:** Client bisa mengirim `userId` orang lain dan menjawab atas nama mereka. Tidak ada verifikasi bahwa socket connection milik user yang bersangkutan.

**Dampak:**
- Spoofing submission user lain
- Manipulasi skor
- Race condition pada multiple submission

**Fix:** Bind userId ke socket session saat join, jangan terima dari client.

### 2.4 🟠 TINGGI — Answer Dikirim ke Client

**File:** `backend/src/Quiz.ts:94-96`
```typescript
IoManager.getIo().to(this.roomId).emit("problem", {
    problem  // ← object Problem termasuk field `answer`!
})
```

**Risk:** Seluruh object `Problem` (termasuk `answer`) dikirim ke semua client. User bisa inspect WebSocket traffic untuk melihat jawaban benar sebelum menjawab.

**Fix:** Strip answer sebelum mengirim ke client:
```typescript
const { answer, submissions, ...safeProblem } = problem;
IoManager.getIo().to(this.roomId).emit("problem", { problem: safeProblem });
```

### 2.5 🟠 TINGGI — Tidak Ada Rate Limiting

**Tidak ada proteksi terhadap:**
- Brute force admin password
- Spam join/submit events
- DoS via rapid socket connections

**Fix:** Implementasi rate limiter middleware untuk Socket.IO.

### 2.6 🟠 TINGGI — User ID Menggunakan Random String Lemah

**File:** `backend/src/Quiz.ts:147-155`
```typescript
genRandonString(length: number) {
    var chars = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ!@#$%^&*()';
    // ...
    result += chars.charAt(Math.floor(Math.random() * charLength));
}
```

**Risk:**
- `Math.random()` bukan CSPRNG — bisa diprediksi
- Karakter `!@#$%^&*()` berpotensi masalah encoding
- Tidak ada collision check

**Fix:** Gunakan `crypto.randomUUID()` atau `nanoid`.

### 2.7 🟡 SEDANG — File-Based Participant Data

**File:** `backend/src/access/ParticipantAccess.ts:69`
```typescript
const markdown = fs.readFileSync(participantFile, "utf8");
```

**Risk:**
- File dibaca dari filesystem — path traversal jika file name bisa dimanipulasi
- Data di-cache di memory (`participantSlotsByNpm`) tanpa invalidation
- Tidak ada hot-reload jika file berubah

### 2.8 🟡 SEDANG — Tidak Ada Input Sanitization

Data dari client (room ID, problem title, description) tidak di-sanitize sebelum disimpan atau di-emit ke client lain. Potensi XSS jika data dirender tanpa escape.

### 2.9 🟡 SEDANG — Session Storage Sensitivity

**File:** `frontend/src/contexts/SocketContext.tsx:38`
```typescript
const SESSION_STORAGE_KEY = 'quizroom:participant-session';
```

Data NPM, userId, dan access info disimpan di localStorage. Jika shared computer digunakan, data peserta bisa terekspos.

## 3. KEKURANGAN & TECHNICAL DEBT

### 3.1 Backend Issues

| # | Issue | Severity | Detail |
|---|-------|----------|--------|
| 1 | **In-Memory State Only** | 🔴 Kritis | Semua quiz state hilang saat server restart. Tidak ada database. |
| 2 | **No TypeScript `any` discipline** | 🟠 Tinggi | Banyak implicit any, terutama di handler params |
| 3 | **Debug logging di production** | 🟡 Sedang | `console.log` tersebar di mana-mana, termasuk password logging |
| 4 | **No graceful shutdown** | 🟡 Sedang | Tidak ada SIGTERM/SIGINT handler — user ter-putus tanpa notifikasi |
| 5 | **`setInterval` memory leak** | 🟠 Tinggi | Quiz constructor buat `setInterval` tapi tidak pernah di-clear |
| 6 | **No error boundary di socket handlers** | 🟡 Sedang | Error di handler bisa crash seluruh server |
| 7 | **Quiz array never cleaned up** | 🟠 Tinggi | `quizes[]` di QuizManager tumbuh terus-menerus |
| 8 | **globalProblemId counter reset** | 🟡 Sedang | Counter reset saat restart, ID bisa collision |
| 9 | **No validation on `next()`** | 🟡 Sedang | Admin bisa spam "next" saat quiz belum selesai |
| 10 | **`start()` tanpa problems** | 🟡 Sedang | Quiz bisa di-start tanpa ada soal |

### 3.2 Frontend Issues

| # | Issue | Severity | Detail |
|---|-------|----------|--------|
| 1 | **Hardcoded backend URL** | 🔴 Kritis | `io('http://localhost:3000')` — tidak bisa deploy |
| 2 | **Monolithic SocketContext** | 🟠 Tinggi | Semua state dan logic dalam satu file 216 baris |
| 3 | **No error boundaries** | 🟠 Tinggi | Socket error bisa crash entire UI |
| 4 | **`any` type pada quizState** | 🟡 Sedang | `quizState: any \| null` — kehilangan type safety |
| 5 | **Race condition di timer** | 🟡 Sedang | `selectedOption` closure di timer bisa stale |
| 6 | **No offline/reconnect handling** | 🟠 Tinggi | Koneksi putus = data hilang, tidak ada re-sync |
| 7 | **Duplicate types** | 🟡 Sedang | `types.ts` di-define di backend DAN frontend secara terpisah |
| 8 | **`onKeyPress` deprecated** | 🟡 Sedang | Harusnya `onKeyDown` |
| 9 | **No loading states** | 🟡 Sedang | Admin panel tidak ada skeleton/spinner |
| 10 | **Admin panel accessible tanpa guard** | 🟠 Tinggi | Route `/admin` bisa diakses siapa saja, auth hanya via socket |

### 3.3 Infrastructure Issues

| # | Issue | Severity | Detail |
|---|-------|----------|--------|
| 1 | **No Docker configuration** | 🟠 Tinggi | Deployment manual, tidak reproducible |
| 2 | **No CI/CD pipeline** | 🟡 Sedang | Build dan deploy manual |
| 3 | **No health check endpoint** | 🟡 Sedang | Tidak bisa monitoring uptime |
| 4 | **No environment configuration** | 🔴 Kritis | Tidak ada `.env` support |
| 5 | **Build output di repo** | 🟡 Sedang | `dist/` folder committed ke git |
| 6 | **No monorepo tooling** | 🟡 Sedang | Backend dan frontend punya `package.json` terpisah tanpa workspace |

### 8.3 🔴 BUG BARU — setTimeout Stacking (Race Condition)

**File:** `backend/src/Quiz.ts:69-79`

```typescript
setActiveProblem(problem: Problem) {
    this.currentState = "question"
    problem.startTime = new Date().getTime();
    problem.submissions = [];
    IoManager.getIo().to(this.roomId).emit("problem", {
        problem: this.sanitizeProblemForClient(problem)
    })
    setTimeout(() => {
        this.sendLeaderboard();
    }, PROBLEM_TIME_S * 1000);
}
```

**Masalah:** Setiap kali `setActiveProblem()` dipanggil, sebuah `setTimeout` baru dibuat **tanpa membatalkan yang sebelumnya**. Skenario:

```
T=0s   → Soal 1 dikirim, setTimeout A dibuat (fire di T=20s)
T=10s  → Admin klik "Next", Soal 2 dikirim, setTimeout B dibuat (fire di T=30s)
T=20s  → setTimeout A FIRE → leaderboard dikirim (tapi soal 2 baru jalan 10 detik!)
T=30s  → setTimeout B FIRE → leaderboard dikirim lagi (double emit)
```

**Dampak:**
- Leaderboard muncul di tengah-tengah soal yang sedang berjalan
- Double/triple leaderboard emission
- Peserta melihat leaderboard saat masih ada waktu menjawab
- Skor bisa tidak akurat karena leaderboard dikirim sebelum waktu habis

**Fix:**
```typescript
private leaderboardTimeout: ReturnType<typeof setTimeout> | null = null;

setActiveProblem(problem: Problem) {
    if (this.leaderboardTimeout) {
        clearTimeout(this.leaderboardTimeout);
    }
    this.currentState = "question";
    problem.startTime = new Date().getTime();
    problem.submissions = [];
    IoManager.getIo().to(this.roomId).emit("problem", {
        problem: this.sanitizeProblemForClient(problem)
    });
    this.leaderboardTimeout = setTimeout(() => {
        this.sendLeaderboard();
        this.leaderboardTimeout = null;
    }, PROBLEM_TIME_S * 1000);
}
```

### 8.4 🔴 BUG BARU — Points Bisa Negatif

**File:** `backend/src/Quiz.ts:189`

```typescript
user.points += (1000 - (500 * (new Date().getTime() - problem.startTime) / (PROBLEM_TIME_S * 1000)));
```

**Masalah:** Formula ini TIDAK memiliki floor (batas bawah). Jika:
- `problem.startTime` belum di-set dengan benar (race condition)
- Ada clock skew antara `setActiveProblem` dan `submit`
- Submit terjadi tepat di boundary waktu

Maka `(new Date().getTime() - problem.startTime)` bisa lebih besar dari `PROBLEM_TIME_S * 1000`, menghasilkan nilai negatif.

**Contoh perhitungan:**
```
Jika elapsed = 25 detik (terlambat 5 detik):
points += 1000 - (500 * 25000 / 20000)
points += 1000 - 625
points += -375  ← NEGATIVE!
```

**Fix:**
```typescript
const elapsed = new Date().getTime() - problem.startTime;
const maxTime = PROBLEM_TIME_S * 1000;
const timeRatio = Math.min(elapsed / maxTime, 1); // Clamp to [0, 1]
const points = Math.max(0, Math.round(1000 - (500 * timeRatio)));
user.points += points;
```

**File:** `frontend/src/contexts/SocketContext.tsx:78-147`

### 8.16 🟡 BUG — `connectionTimeout` Tidak Di-Reset

**File:** `frontend/src/components/Quiz.tsx:29-41`

```typescript
useEffect(() => {
    const timeout = setTimeout(() => {
        if (!quizState) {
            setConnectionTimeout(true);
        }
    }, 5000);

    if (quizState) {
        setConnectionTimeout(false);
        clearTimeout(timeout);
    }

    return () => clearTimeout(timeout);
}, [quizState]);
```

**Masalah:** Dependency array hanya `[quizState]`. Jika `quizState` berubah dari `null` ke `{ type: 'not_started' }`, timeout di-clear. Tapi jika quizState berubah lagi (misal: ke `question`), effect re-runs dan timeout baru dibuat. Ini sebenarnya OK karena `quizState` selalu truthy setelah join.

**TAPI:** Jika user di-kick dari room (misal: access expired), `quizState` bisa kembali `null`, dan timeout akan fire lagi. Ini sebenarnya behavior yang benar.

**Verdict:** Minor — tidak perlu fix, tapi perlu dokumentasi.

### 8.20 🟡 ISSUE — Tidak Ada Quiz Ended Handler di Frontend

// Quiz.ts
next() {

**Fix (Frontend):**

// SocketContext.tsx
newSocket.on('quizEnd', (data) => {
